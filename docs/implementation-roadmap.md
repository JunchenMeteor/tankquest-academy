# 实施路线图

## 1. 开发原则

- 小 PR。
- 每个 PR 一个主题。
- 先闭环，再扩展。
- 不把 UI、后端、AI、资源管线混成一个巨大提交。

## 2. 阶段状态

阶段状态分为“工程验收”和“产品体验验收”。前者证明规则、测试、构建和部署可靠，后者证明内容密度、操作反馈、重复游玩和家庭理解度达到产品目标；不能再用单一“已完成”替代两种结论。

| 阶段 | 工程状态 | 产品体验状态 | 说明 |
| --- | --- | --- | --- |
| Phase 1 工程 MVP | 已验收 | 待家庭验证 | 三关卡选择、驾驶射击、3 道数学任务、后端结算、奖励升级和持久化闭环 |
| Phase 1.5 战斗收口 | 已验收 | 待 P5.1 收口 | 生命与伤害、实体碰撞、撞击公式、敌方坦克分级、差异化地图和无死路结算 |
| Phase 1.6 命中判定收口 | 已验收 | 待 P5.1 收口 | 确定性穿深、方向装甲、跳弹/未击穿/击穿反馈和双方统一规则 |
| Phase 2 养成与学习记录 | 已验收 | 内容深度不足 | 三辆坦克、涂装、学习记录、家长报告和三类学科链路可用，但正式题库仍接近纵向切片 |
| Phase 3 AI 学习教练 | 已验收 | 儿童侧感知有限 | 结构化 AI、错题解释、自适应建议、家长摘要和安全草稿已通过工程边界验收 |
| Phase 4 2.5D Web 升级 | 已验收 | 待视觉体验收口 | Phaser 2.5D、资源清单、主题场景、触控和安全 PWA 壳已通过工程验收 |
| Phase 5 Windows 技术预览 | 已验收 | 不改变现有内容 | Tauri 2 Windows 壳、平台 adapter、启动降级和 Windows 构建烟测已通过工程验收 |
| Phase 5.1 内容与产品体验 | 待实施 | 待验收 | 扩充正式内容、关卡目标、敌人行为、成长反馈、视听表现并完成家庭试用 |
| Phase 6+ | 待评估 | 待评估 | 完整 low-poly 3D、多人玩法和移动端分别独立评估，不用于替代 P5.1 产品收口 |

Phase 1 的逐项结论和非阻塞项见 `phase-one-mvp-acceptance.md`。家庭试用发现的问题进入独立 Issue，不回填到已冻结的工程 MVP 范围。

Phase 2 的逐项结论和证据见 `phase-two-acceptance.md`。正式多家庭鉴权和真实家庭可用性验证仍属于后续工作，不阻塞 v0.2.0 工程发布。

Phase 3 的交付边界见 `phase-three-scope.md`，验收证据见 `phase-three-acceptance.md`，工程版本为 v0.3.0。AI 只提供建议、解释和草稿，主后端继续裁决正确性、奖励、升级、难度和内容发布。

Phase 4 的交付边界见 `phase-four-scope.md`，验收证据见 `phase-four-acceptance.md`，工程版本为 v0.4.0。本阶段选择 Phaser 2.5D 等距表现并保留现有 Arcade Physics，以 PWA 作为客户端包装基础；完整 3D 和原生客户端不进入 v0.4.0。

Phase 5 的交付边界见 `phase-five-scope.md`，验收证据见 `phase-five-acceptance.md`，工程版本为 v0.5.0。本阶段只实现复用现有 Web 产品的 Windows Tauri 技术预览；完整 3D、多人、移动端、代码签名、商店发布和自动更新继续后置。

Phase 5.1 的交付边界见 `phase-five-one-product-experience-scope.md`，目标版本为 v0.6.0。本阶段先解决当前纵向切片在内容量、关卡变化、敌人感知、成长反馈和视听表现上的产品缺口，再评估完整 3D、多人和移动端。

Phase 1.6 和 Phase 2 的交付边界分别见 `phase-one-six-combat-scope.md` 和 `phase-two-scope.md`。Phase 1.6 完成后不再追加新的 1.x 工程阶段。

## 3. PR 拆分建议

### PR 1: 项目脚手架

- 前端 Vite + TypeScript。
- 后端基础服务。
- 数据库连接。
- 基础 CI。

### PR 2: 核心数据模型

- 用户/孩子。
- 坦克。
- 关卡。
- 题库。
- 对局。

### PR 3: 游戏 MVP

- Phaser 初始化。
- 坦克移动。
- 炮塔旋转。
- 射击。
- 碰撞。

### PR 4: 对局 API

- 创建对局。
- 提交答案。
- 提交事件。
- 结束结算。

### PR 5: 学习任务

- 数学题。
- 正确答案弹药箱。
- 错题解释模板。

### PR 6: 奖励和升级

- 星级。
- 零件。
- 坦克升级。

### PR 7: 后台最小版

- 题库管理。
- 关卡管理。
- 坦克管理。

### PR 8: 家长控制

- 孩子档案。
- 游玩时长。
- 难度范围。

### PR 9: AI 服务接入

- FastAPI AI 服务。
- 出题草稿。
- 错题解释。
- 安全校验。

### PR 10: 资源管线

- 资源 manifest。
- 模型/贴图加载。
- hash 校验。
- CDN/静态资源适配。

## 4. 每个 PR 必须说明

```text
改动范围
数据来源
新增 API
新增表
测试结果
风险
不做什么
```

## 5. 验证命令

具体命令按实际项目补充，但必须包含：

```text
lint
typecheck
unit test
contract test
build
e2e smoke
```
